## 一、Agent系统设计

### 1.1 角色定义（6种Agent）

| 角色 | 核心职责 | 输入 | 输出 |
|------|----------|------|------|
| **Product Designer** | 需求分析、产品定位、PRD撰写 | 用户原始需求 | PRD文档 |
| **UI Designer** | 界面设计、交互设计 | PRD（UI部分） | Figma设计文件 |
| **Program Architect** | 技术选型、架构设计、API定义 | PRD（技术部分） | 架构文档 + API接口文档 |
| **Developer** | 代码实现、功能开发 | API文档 + 架构文档 + UI设计 | 可运行代码 |
| **QA Engineer** | 测试用例设计、质量把控 | PRD + API文档 + UI设计 | 测试用例文档 |
| **Code Reviewer** | 代码评审、规范把控 | 开发者提交的代码 | Review意见 |
| **Product Manager** | 项目管理、进度跟踪、任务协调、确定项目路径 | 项目整体情况 | 任务看板状态、项目报告 |
| **DevOps** | 环境配置、CI/CD、部署 | 技术选型 + 项目路径（PM指定） | 开发环境 + 测试环境 + 验收环境（三套隔离环境） |

### 1.2 测试执行分工

| 测试类型 | 执行者 | 说明 |
|----------|--------|------|
| 单元测试 | Developer | 开发过程中编写并执行 |
| 集成测试 | Developer | 基于API文档进行接口联调 |
| 功能测试 | QA Engineer | 使用QA编写的测试用例执行 |

> **说明**：QA Engineer 负责设计测试用例，Developer 负责执行单元测试和集成测试，功能测试由 QA Engineer 执行。

## 二、用户确认节点

### 2.1 确认节点定义

| 节点 | 执行者 | 确认内容 | 触发条件 |
|------|--------|----------|----------|
| **PRD确认** | 用户 | 产品需求是否准确、完整 | Product Designer 完成 PRD 后 |
| **UI确认** | 用户 | 界面设计是否符合预期 | UI Designer 完成设计后（可选） |
| **验收确认** | 用户 | 功能是否满足需求 | 开发完成 + 测试通过后 |

### 2.2 测试失败处理流程

| 阶段 | 执行者 | 处理方式 |
|------|--------|----------|
| **单元测试失败** | Developer | 立即修复，重新执行单元测试，直到通过 |
| **集成测试失败** | Developer | 检查接口兼容性，修复后重新联调 |
| **功能测试失败** | QA Engineer | 记录缺陷，返回Developer修复，重新提交测试 |

> **循环规则**：测试失败后返回上一环节修复，修复完成后重新执行测试，形成闭环直到通过。

### 2.3 验收标准定义

| 验收节点 | 验收内容 | 通过条件 |
|----------|----------|----------|
| **PRD确认** | 需求完整性、逻辑一致性 | 用户签字确认/明确回复"同意" |
| **UI确认** | 交互流程、视觉规范 | 用户确认/48h内无异议 |
| **代码Review** | 代码规范、架构一致性 | Reviewer通过 |
| **功能测试** | 测试用例100%通过 | QA Engineer签署测试报告 |
| **最终验收** | 功能可用、无阻塞缺陷 | 用户确认"可交付"

1. **PRD确认**  
   - Product Designer 输出 PRD 后，主动征询用户修改意见
   - 用户确认无异议后，才能进入下一步（UI设计/架构设计）
   - 如有修改，返回 Product Designer 调整

2. **UI确认（可选）**  
   - 复杂项目可在 UI Designer 完成设计后增加用户确认环节
   - 简单项目可跳过，直接进入开发

3. **验收确认**  
   - 开发完成 + 测试通过后，由 PM 协调用户验收
   - 用户确认后项目结束

## 三、任务依赖与执行规则

### 3.1 任务依赖关系

| 任务 | 依赖任务 | 说明 |
|------|----------|------|
| UI设计 | PRD完成 | 必须有PRD后才能开始UI设计 |
| 架构设计 | PRD完成 | 必须有PRD后才能开始技术架构 |
| API文档 | 架构设计完成 | API基于架构定义 |
| 环境配置 | 架构设计完成 + PM指定项目路径 | DevOps获取技术栈后创建三套环境 |
| 开发环境 | 环境配置完成 | Developer开始开发 |
| 测试环境 | 环境配置完成 | QA开始测试 |
| 验收环境 | 环境配置完成 | 用户验收 |
| CI/CD | 环境配置完成 | 自动构建/测试/部署 |
| 测试用例 | PRD + UI设计 | 需要完整需求、设计、接口 |
| 代码开发 | API文档 + UI设计完成 + 开发环境就绪 | 必须有明确API + UI设计才能写代码 |
| 代码Review | 代码开发完成 | 必须有代码才能Review |
| 功能测试 | 代码Review通过 + 单元/集成测试通过 + 测试环境就绪 | 必须通过Review才能测试 |
| 项目验收 | 功能测试通过 + 用户确认 + 验收环境就绪 | 需要功能实现并通过测试 |

### 3.2 并行与串行规则

- **可并行**：UI设计 ↔ 架构设计 ↔ 环境配置（独立进行）
- **可并行**：API文档 ↔ 测试用例设计（独立进行）
- **串行**：PRD → 用户确认 → UI设计/架构设计 → 环境配置 → API → 开发 → 功能测试 → 验收

#### 并行协调机制

| 场景 | 协调方式 |
|------|----------|
| UI设计 + 架构设计并行 | 双方各自独立输出，通过PM协调汇总 |
| API文档 + 测试用例并行 | 双方基于PRD各自独立工作，最终交叉校验 |
| 多模块并行开发 | 按模块分配Developer，独立开发后集成联调 |

> **注意**：并行任务完成后，需等待双方均就绪才能进入下一环节（由PM确认）

## 四、PM 职责细化

### 4.1 项目管理职责

1. **项目启动**
   - 用户提出需求时，立即创建工作流
   - 创建 Product Designer 任务，启动需求分析

2. **工作流创建**
   - 定义项目阶段（需求 → 设计 → 开发 → 测试 → 验收）
   - 设置阶段门禁（Gate）

3. **任务管理**
   - 创建具体任务
   - 分配给对应Agent
   - 定义优先级（P0/P1/P2）
   - 设置截止时间

4. **依赖管理**
   - 梳理任务间依赖
   - 协调阻塞问题

5. **进度跟踪**
   - 更新任务状态
   - 生成进度报告
   - 识别风险并预警

6. **项目环境规划（DevOps协作）**
   - 确定项目开发路径（代码仓库位置）
   - 确定文档存储位置
   - 将项目路径传递给 DevOps 创建环境

### 4.2 不属于PM的职责

- 不负责撰写PRD（Product Designer）
- 不负责写代码（Developer）
- 不负责测试（QA Engineer）

## 五、信息传递规范

### 5.1 Agent协作通信机制

#### 5.1.1 通信方式

| 场景 | 通信方式 | 说明 |
|------|----------|------|
| 任务分配 | `sessions_spawn` | PM创建子任务，指定Agent角色 |
| 上下文传递 | 共享workspace | 上游产物写入workspace，下游读取 |
| 状态同步 | Bitable任务看板 | 任务状态实时更新到多维表格 |
| 结果汇报 | 会话消息 | Agent完成任务后返回结果摘要 |

#### 5.1.2 上下文传递规则

| 上游产物 | 传递方式 | 下游接收者 |
|----------|----------|------------|
| 项目路径（PM指定） | workspace配置 + spawn传递 | DevOps |
| PRD文档 | 写入workspace + spawn传递 | UI Designer, Program Architect, QA |
| Figma设计 | 链接/文件路径 | Developer, QA |
| 架构文档 | 写入workspace + spawn传递 | DevOps, Developer |
| API文档 | 写入workspace + spawn传递 | Developer, QA |
| 测试用例 | 写入workspace + spawn传递 | Developer |
| 代码仓库 | Git共享/workspace目录 | DevOps, QA, PM |

#### 5.1.3 spawn使用规范

```typescript
// PM分配任务示例
sessions_spawn({
  agentId: "product-designer",
  task: "基于用户需求撰写PRD",
  runtime: "subagent",
  mode: "run"  // 一次性任务
})

// 持续性任务使用 session 模式
sessions_spawn({
  agentId: "developer",
  task: "实现API对应的代码模块",
  runtime: "subagent",
  mode: "session"  // 持续对话
})
```

## 七、任务状态管理

### 7.1 Bitable任务看板设计

| 字段 | 类型 | 说明 |
|------|------|------|
| 任务名称 | 文本 | 具体任务描述 |
| 角色 | 单选 | Product Designer / UI Designer / Program Architect / Developer / Code Reviewer / QA Engineer / DevOps / PM |
| 状态 | 单选 | 待处理 / 进行中 / 已完成 / 阻塞 / 环境就绪 / 已验收 |
| 优先级 | 单选 | P0 / P1 / P2 |
| 负责人 | 用户 | 分配的Agent或人 |
| 依赖任务 | 关联 | 上游任务ID |
| 截止时间 | 日期 | 计划完成时间 |
| 产出物 | URL | 文档/代码链接 |

### 7.2 状态更新规则

| 事件 | 状态变更 | 触发者 |
|------|----------|--------|
| 任务创建 | → 待处理 | PM |
| 开始执行 | → 进行中 | 承担该角色的Agent |
| 产出完成 | → 已完成 | Agent |
| 环境配置完成 | → 环境就绪 | DevOps（特指环境配置任务） |

| 问题解决 | → 进行中 | PM/Agent |
| 用户确认 | → 已验收 | 用户 |

## 八、环境管理约束（DevOps核心原则）

> **关键约束**：任何与环境配置有关的事情都必须由 DevOps 负责，确保开发、测试、验收三套环境隔离运行。

### 8.1 环境隔离原则

| 环境 | 用途 | 隔离要求 |
|------|------|----------|
| **开发环境** | Developer 日常开发、单元测试 | 本地容器/虚拟环境，不污染全局 |
| **测试环境** | QA 功能测试、集成测试 | 独立容器/独立服务器，与开发隔离 |
| **验收环境** | 用户验收、与生产一致 | 独立环境，确保与生产一致 |

### 8.2 环境管理规则

- **局部环境**：使用容器（Docker）或虚拟环境隔离，不污染系统全局
- **三套隔离环境**：开发/测试/验收环境完全隔离，互不影响
- **环境即代码**：所有环境配置必须版本化管理（Dockerfile/ docker-compose.yml）
- **PM 指定路径**：项目路径和文档存储位置由 PM 在创建工作流时确定，传递给 DevOps
- **环境就绪**：环境配置完成后的状态为"环境就绪"，下游任务才能启动
